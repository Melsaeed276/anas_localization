## Dependencies & Execution Order

### Phase Dependencies

- **Setup (Phase 1)**: No dependencies.
- **Foundational (Phase 2)**: Depends on Setup completion and blocks all user stories.
- **User Story 1 (Phase 3)**: Depends on Foundational and is the MVP.
- **User Story 2 (Phase 4)**: Depends on Foundational and can start after US1 public configuration shape is stable.
- **User Story 3 (Phase 5)**: Depends on Foundational and benefits from US2 cache snapshots, but its merge policy can be tested independently.
- **Polish (Phase 6)**: Depends on all implemented user stories.

### User Story Dependencies

- **US1 (P1)**: Required MVP; no dependency on US2 or US3.
- **US2 (P2)**: Requires foundational types and uses the public remote service shape introduced by US1.
- **US3 (P3)**: Requires cache snapshots from foundational/US2 work and integrates with the existing `LocalizationService` merge path.

### Within Each User Story

- Write tests first and run the focused test file to confirm failure.
- Implement only the code needed for those tests.
- Run the focused test file again.
- Format touched files before moving to the next story.

## Implementation Strategy

### MVP First

1. Complete Phase 1 and Phase 2.
2. Complete Phase 3 only.
3. Validate that local-only apps behave exactly as before and configured remote startup does not block local rendering.
4. Stop and review before adding manual update behavior.

### Incremental Delivery

1. Add foundational API types and contract tests.
2. Deliver US1 for opt-in, non-blocking configuration.
3. Deliver US2 for manual checks, cache writes, retry, queueing, logging, and metrics.
4. Deliver US3 for protected merge behavior.
5. Run full gates and update docs.

## Independent Test Criteria

- **US1**: `flutter test test/features/localization/remote_localization_integration_test.dart` proves no-remote and remote-configured startup remain local-first.
- **US2**: Focused cache, repository, coordinator, and use-case tests prove manual checks apply only valid newer payloads and return structured results.
- **US3**: Merge policy and integration tests prove protected keys win and metadata is not exposed.

## Notes For Smaller Implementation Models

- Do tasks strictly in numeric order unless a human assigns parallel work.
- Do not skip failing-test steps for test tasks.
- Do not infer new API names when a task names a class or file; use the names in this file and the contract in `specs/009-remote-localization/contracts/remote-localization-api.md`.
- Keep V1 runtime-only: do not add CLI remote push, Catalog remote sync, remote history, or app asset mutation.
